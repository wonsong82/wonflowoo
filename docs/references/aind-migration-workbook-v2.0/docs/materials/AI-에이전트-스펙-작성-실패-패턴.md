# AI 에이전트 스펙 작성, 2,500개 파일이 알려준 실패 패턴

> **출처**: [AI Sparkup](https://aisparkup.com/posts/9476) | 원문: [O'Reilly Radar](https://www.oreilly.com/radar/how-to-write-a-good-spec-for-ai-agents/)

## 핵심 요약

Google Chrome 엔지니어링 디렉터 Addy Osmani가 GitHub의 2,500개 에이전트 설정 파일 분석 결과를 토대로 정리한 AI 코딩 에이전트 스펙 작성 원칙. 결론: **"대부분의 에이전트 파일은 너무 모호하기 때문에 실패한다."**

---

## 스펙이 실패하는 두 가지 함정

### 1. 모호함
- ❌ "유용한 코딩 어시스턴트처럼 행동해"
- ✅ "React 컴포넌트 테스트를 작성하는 테스트 엔지니어이며, 소스 코드는 절대 수정하지 않는다"
- 역할과 제약을 명확히 정의해야 작동한다

### 2. 과부하 — "지시의 저주 (Curse of Instructions)"
- 지시사항이 많아질수록 모델이 앞 몇 가지만 따르고 나머지를 흘려버림
- 50페이지짜리 문서를 통째로 넣으면 품질 개선이 아니라 역효과
- RFC 수준의 완벽한 스펙 → 컨텍스트가 커지면 모델이 망가짐

---

## 성공하는 스펙의 6가지 핵심 영역

| 영역 | 내용 |
|------|------|
| **Commands** | 도구 이름이 아닌 실제 명령어 — `npm test`, `pytest -v` |
| **Testing** | 프레임워크, 테스트 파일 위치, 커버리지 기준 |
| **Project structure** | `src/`는 앱 코드, `tests/`는 단위 테스트 등 명시적으로 |
| **Code style** | 세 문단 설명보다 실제 코드 한 줄이 낫다 |
| **Git workflow** | 브랜치 네이밍, 커밋 메시지 형식, PR 요건 |
| **Boundaries** | 에이전트가 절대 건드리면 안 되는 것들 |

> "시크릿 절대 커밋 금지" — 2,500개 파일 중 가장 많이 등장한 제약

---

## 3계층 경계 설정 (핵심)

단순한 금지 목록보다 훨씬 효과적인 구조:

- ✅ **Always**: 묻지 않고 해도 되는 것 — "커밋 전 항상 테스트 실행"
- ⚠️ **Ask first**: 사람 승인이 필요한 것 — "데이터베이스 스키마 변경 전 확인"
- 🚫 **Never**: 절대 불가 — "시크릿이나 API 키 절대 커밋 금지"

이 구조는 에이전트가 안전하게 진행할 영역과 멈춰야 할 영역을 명확히 구분해, 단순 규칙 목록과 달리 상황에 맞는 판단을 가능하게 한다.

## 큰 스펙은 잘게 나눠라

- `SPEC_backend.md`, `SPEC_frontend.md`처럼 분리
- 백엔드 작업 시 → 백엔드 스펙만, 프론트엔드 작업 시 → 프론트엔드 스펙만 제공
- 각 작업마다 필요한 컨텍스트만

### Plan Mode 활용
- Claude Code에서 **Shift+Tab**으로 활성화
- 에이전트가 코드베이스 분석 + 계획 수립, 실제 코드는 작성 안 함
- 계획이 충분히 정교해진 후 실행 모드 전환 → 스펙 사전 검증

---

## 스펙은 살아있는 문서

- 에이전트가 잘못 이해하거나 요구사항이 바뀔 때마다 업데이트
- Git으로 버전 관리 → 에이전트도 히스토리 참조 가능
- **"팀의 PRD처럼 모두가(사람이든 AI든) 참조하는 단일 진실 공급원"** — Osmani
