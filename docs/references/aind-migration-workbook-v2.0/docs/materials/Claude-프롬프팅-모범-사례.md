# Claude 프롬프팅 모범 사례

> **출처**: [Anthropic Platform Docs](https://platform.claude.com/docs/ko/build-with-claude/prompt-engineering/claude-prompting-best-practices)

## 핵심 요약

Claude Opus 4.6, Sonnet 4.5, Haiku 4.5 등 최신 모델을 위한 프롬프트 엔지니어링 가이드. 이전 모델보다 더 정밀한 지시 따르기를 위해 훈련된 모델들의 특성을 반영하여, 명확한 지시, 맥락 제공, 장기 추론, 도구 사용 패턴 등 다양한 기법을 다룬다.

---

## 일반 원칙

### 1. 명확하고 구체적인 지시사항

- 원하는 출력을 구체적으로 설명
- "기대 이상"의 동작을 원한다면 명시적으로 요청
- 예: "분석 대시보드를 만들어주세요" → "가능한 한 많은 관련 기능과 인터랙션을 포함하고, 완전한 기능을 갖춘 구현을 만들어주세요"

### 2. 맥락(동기) 제공으로 성능 향상

- 왜 그런 동작이 중요한지 설명하면 더 타겟팅된 응답 가능
- 예: "줄임표를 사용하지 마세요" → "TTS 엔진이 줄임표를 발음할 수 없으므로 사용하지 마세요"

### 3. 예시와 세부사항에 주의

- Claude는 세부사항과 예시에 세심한 주의를 기울임
- 예시가 원하는 동작과 일치하는지 확인

## 장기 추론 및 상태 추적

### 컨텍스트 인식

- Opus 4.6, 4.5 모델들은 남은 컨텍스트 윈도우(토큰 예산)를 추적 가능
- 에이전트 하네스에서 사용 시, 컨텍스트 자동 압축 안내를 프롬프트에 추가 권장

```text
컨텍스트 윈도우는 제한에 가까워지면 자동으로 압축되어, 중단한 곳에서 무기한으로 작업을 계속할 수 있습니다.
토큰 예산 우려로 인해 작업을 일찍 중단하지 마세요.
```

### 다중 컨텍스트 윈도우 워크플로우

1. **첫 번째 윈도우**: 프레임워크 설정 (테스트 작성, 설정 스크립트)
2. **구조화된 테스트**: `tests.json` 같은 형식으로 추적
3. **편의 도구 설정**: `init.sh` 같은 스크립트로 반복 작업 방지
4. **새로 시작 vs 압축**: 완전히 새 윈도우에서 시작하는 것도 고려 (로컬 파일시스템에서 상태 발견에 탁월)
5. **검증 도구 제공**: Playwright MCP, 컴퓨터 사용 기능 등
6. **컨텍스트 완전 사용 장려**: 작업 완료까지 체계적으로 계속

### 상태 관리 모범 사례

- 상태 데이터: JSON 등 구조화된 형식
- 진행 노트: 자유 형식 텍스트
- Git: 상태 추적에 특히 효과적
- 점진적 진행 강조

## 커뮤니케이션 스타일

최신 모델의 특성:
- 더 직접적이고 근거 있는 진행 보고
- 더 대화적이고 유창한 톤
- 덜 장황 (요청 없으면 상세 요약 생략 가능)

### 장황함 균형 맞추기

도구 사용 후 업데이트를 원한다면:
```text
도구 사용을 포함하는 작업을 완료한 후, 수행한 작업에 대한 간단한 요약을 제공하세요.
```

## 도구 사용 패턴

### 행동 vs 제안

- "변경 사항을 제안해 줄 수 있나요?" → 제안만 할 수 있음
- "이 함수의 성능을 개선하도록 변경하세요" → 실제로 변경 수행

### 적극적 행동 유도

```text
<default_to_action>
기본적으로 변경 사항을 제안만 하지 말고 구현하세요. 사용자의 의도가 불분명한 경우,
가장 유용할 것 같은 행동을 추론하고 진행하며, 도구를 사용하여 누락된 세부사항을 발견하세요.
</default_to_action>
```

### 보수적 행동 유도

```text
<do_not_act_before_instructions>
변경을 하라는 명확한 지시가 없는 한 구현이나 파일 변경에 뛰어들지 마세요.
사용자의 의도가 모호한 경우, 정보를 제공하고 권장 사항을 제공하는 것을 기본으로 하세요.
</do_not_act_before_instructions>
```

### 도구 트리거링 주의사항

- Opus 4.5/4.6은 시스템 프롬프트에 더 민감
- "중요: 반드시 이 도구를 사용해야 합니다" → "이 도구를 사용하세요" (과다 트리거링 방지)

## 자율성과 안전성

되돌리기 어려운 행동 전에 확인 유도:

```text
파일 편집이나 테스트 실행 같은 되돌릴 수 있는 행동은 취하도록 권장되지만,
되돌리기 어렵거나 공유 시스템에 영향을 미치는 행동에 대해서는 사용자에게 확인하세요.

확인 필요 예시:
- 파괴적 작업: 파일/브랜치 삭제, rm -rf
- 되돌리기 어려운 작업: git push --force, git reset --hard
- 다른 사람에게 보이는 작업: 코드 푸시, PR 댓글, 메시지 보내기
```

## 과도한 사고와 철저함 조절

Opus 4.6은 이전 모델보다 훨씬 많은 사전 탐색 수행.

조절 방법:
- 포괄적 기본값 → 타겟팅된 지시로 교체
- "의심스러우면 [도구]를 사용하세요" 같은 과도한 프롬프팅 제거
- `effort` 매개변수로 조절

```text
문제에 접근하는 방법을 결정할 때, 접근 방식을 선택하고 전념하세요.
추론에 모순되는 새로운 정보가 없는 한 결정을 재검토하지 마세요.
```

## 응답 형식 제어

1. **해야 할 것을 말하기**: "마크다운 쓰지 마세요" → "매끄럽게 흐르는 산문 단락으로 구성하세요"
2. **XML 형식 표시자**: `<smoothly_flowing_prose_paragraphs>` 태그 활용
3. **프롬프트 스타일 = 출력 스타일**: 프롬프트에서 마크다운 제거하면 출력에서도 감소
4. **상세 서식 지시**: 마크다운 최소화를 위한 구체적 가이드라인 제공

## 서브에이전트 오케스트레이션

최신 모델은 향상된 네이티브 서브에이전트 오케스트레이션 능력 보유.

```text
작업이 병렬로 실행 가능하거나, 격리된 컨텍스트가 필요하거나,
독립적인 작업 스트림을 포함할 때 서브에이전트를 사용하세요.
간단한 작업, 순차적 작업, 단일 파일 편집은 직접 작업하세요.
```

주의: Opus 4.6은 서브에이전트 과다 사용 경향 있음 (직접 grep이 더 빠른 경우에도 서브에이전트 생성)

## 적응형 사고 (Adaptive Thinking)

Opus 4.6의 새로운 사고 모드:
- `thinking: {type: "adaptive"}` — Claude가 언제 얼마나 사고할지 동적 결정
- `effort` 매개변수로 사고 깊이 제어 (max, high, medium, low)
- 확장 사고(budget_tokens)보다 안정적으로 더 나은 성능

```python
# 이전 (확장 사고)
thinking={"type": "enabled", "budget_tokens": 32000}

# 이후 (적응형 사고)
thinking={"type": "adaptive"}
output_config={"effort": "high"}
```

## 기타 실용 팁

### 병렬 도구 호출 최적화
- 독립적인 도구 호출은 병렬 실행
- `<use_parallel_tool_calls>` 프롬프트로 ~100% 병렬화 달성 가능

### 파일 생성 관리
- 임시 파일 사용 후 정리하도록 지시

### 과도한 엔지니어링 방지
- 요청된 것 이상의 기능/리팩토링/개선 금지
- 변경하지 않은 코드에 독스트링/주석 추가 금지
- 일회성 작업을 위한 헬퍼/추상화 생성 금지

### 프론트엔드 디자인
- "AI 슬롭" 미학 피하기 (Inter/Roboto, 보라색 그라디언트 등)
- 독특한 폰트, 일관된 테마, 애니메이션/마이크로인터랙션 활용

### 환각 최소화
```text
<investigate_before_answering>
열어보지 않은 코드에 대해 절대 추측하지 마세요.
사용자가 특정 파일을 참조하면, 답변하기 전에 반드시 파일을 읽어야 합니다.
</investigate_before_answering>
```

### LaTeX 출력 제어
- Opus 4.6은 수학 표현에 기본 LaTeX 사용
- 일반 텍스트 원하면 명시적 지시 필요

## 마이그레이션 체크리스트 (이전 모델 → 4.6)

1. 원하는 동작을 구체적으로 설명
2. 수식어로 품질/세부사항 향상 장려
3. 특정 기능(애니메이션, 인터랙션 등) 명시적 요청
4. 사고 구성: `budget_tokens` → `adaptive` + `effort`
5. 미리 채워진 응답(prefill) → 시스템 프롬프트 지시로 대체
6. 안티 레이지니스 프롬프팅 줄이기 (Opus 4.6은 훨씬 능동적)
