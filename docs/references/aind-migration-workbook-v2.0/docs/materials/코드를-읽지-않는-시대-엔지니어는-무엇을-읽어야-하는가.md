# 코드를 읽지 않는 시대, 엔지니어는 무엇을 읽어야 하는가

> **출처**: [GeekNews](https://news.hada.io/topic?id=26874)

---

## 핵심 요약

AI로 코딩한 개발자는 퀴즈 점수가 17% 낮다 — 하지만 진짜 문제는 AI 사용 여부가 아니라 **사용 방식**이다. AI를 코드 생성기로 쓰면 가장 빠르지만 가장 적게 배우고, 개념만 물어보고 직접 짜면 느리지만 훨씬 많이 배운다. AI 시대에 엔지니어는 코드 대신 스펙·테스트·아키텍처를 읽는 역할로 전환되고 있으며, 결국 거울에 비출 내공이 있는 사람이 AI를 잘 쓴다.

---

## 본문

### 1. 핵심 연구 결과 (Anthropic)

- AI로 코딩 과제를 완료한 개발자들의 퀴즈 점수가 **17% 낮음**
- 문제는 AI 사용 자체가 아니라 **사용 방식**

| 방식 | 속도 | 학습량 |
|------|------|--------|
| 코드 통째로 맡김 | 가장 빠름 | 가장 적음 |
| 개념만 물어보고 직접 코딩 | 느림 | 월등히 높음 |

---

### 2. 엔지니어 역할의 변화

- **Ben Shoemaker**: "더 이상 코드를 한 줄씩 읽지 않는다. 스펙, 테스트, 아키텍처를 읽는다." → **하네스 엔지니어링**이라는 새로운 접근
- **OpenAI Codex 팀**: 엔지니어 3명이 에이전트만으로 100만 줄 코드를 만들어 수백 명이 쓰는 제품 완성. 투자한 건 코드가 아니라 **문서, 린터, 테스트 인프라**
- **Evan Armstrong**: 코드 생성은 상품화됐지만, 프로덕션에서 코드를 거버닝하는 **"컨텍스트 레이어"**는 상품화되지 않았다
- **Steve Yegge**: "손코딩의 시대는 끝났다." AI 채택 8단계 제시 — Level 4부터 diff를 안 보기 시작, Level 8에서 에이전트 오케스트레이터 직접 구축

---

### 3. 복리 게임의 함정

- **Kent Beck**: 스펙 기반 개발 뒤에는 "결승선 게임"(X에 도달하면 끝) 가정이 숨어있다
- 실제 소프트웨어 개발은 **"복리 게임"** — 오늘의 아키텍처가 6개월 뒤 가능성을 열거나 닫는다
- "더 나은 AGENTS.md로는 복리 게임을 이길 수 없다" — 시스템이 복리로 쌓이도록 설계하는 게 핵심

---

### 4. AI는 거울이다

- **Jeremy Utley (스탠포드)**: "게으르고 싶은 사람에게는 게으름을, 예리해지고 싶은 사람에게는 날카로움을 도와준다"
- TDD/DDD 배경이 있으면 AI에게 그렇게 지시 가능. "그냥 만들어줘"라고 던지면 구조가 엉망인 코드 → **AI가 멍청한 게 아니라 내가 신경 안 쓴 부분을 AI도 신경 안 쓸 뿐**
- **Berkeley 연구**: AI가 비개발자의 코딩을 가능하게 했지만, 결국 엔지니어가 동료의 AI 코드를 리뷰·수정하는 데 더 많은 시간을 쓰게 됨
- **"드라큘라 이펙트"**: 바이브 코딩 전속력 시 생산적인 시간은 하루 3시간이 한계

---

### 5. 그래서 어떻게 쓸 것인가

- AI에게 정답을 요구하지 말고 **대화**하라 → AI가 나에게 질문하게 하라
- **음성 입력 추천**: 타이핑 = 키워드 모드, 음성 = 대화 모드
- **컨텍스트 엔지니어링**: AGENTS.md에 아키텍처 결정 이유, 컨벤션, 도메인 용어를 명시하면 AI가 일관된 코드 생성
- **Kent Beck**: "피처만큼 퓨처스(다음에 구현할 수 있는 것들의 집합)에 투자하라"

---

### 6. 변하지 않는 것들

- 코드를 매 줄 읽을 필요는 줄지만, **읽을 줄 아는 능력은 오히려 더 중요해졌다**
- 모든 테스트가 통과하는데 제품이 이상할 때, AI가 "문제없다"는데 버그가 있을 때 → 결국 직접 읽어야 하는 순간이 온다
- **"읽을 줄 아는데 안 읽는 것"과 "못 읽는 것"은 완전히 다른 이야기**
- 거울에 비출 게 있는 사람이 되는 것 — 그게 이 시대 엔지니어의 본질
