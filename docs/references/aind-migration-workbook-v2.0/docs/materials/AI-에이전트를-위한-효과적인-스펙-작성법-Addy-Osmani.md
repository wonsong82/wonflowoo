# AI 에이전트를 위한 효과적인 스펙 작성법

> **출처**: [GeekNews #25917](https://news.hada.io/topic?id=25917) / 원글: [Addy Osmani - Substack](https://addyo.substack.com/p/how-to-write-a-good-spec-for-ai-agents)

---

## 핵심 요약

AI 에이전트에게 복잡한 작업을 맡길 때 생산성 차이를 만드는 것은 **명확하고 구조화된 SPEC**이다. 나쁜 SPEC은 AI 드리프트와 반복 실패를 낳고, 좋은 SPEC은 살아있는 문서(SPEC.md)로 지속 업데이트된다. 핵심 원칙: **먼저 계획 → 작게 쪼개 → 반복 검증**.

---

## 5가지 핵심 원칙

### 원칙 1: 고수준 스펙으로 시작 → AI가 상세 확장

- 처음부터 과도한 스펙을 작성하는 대신, **명확한 목표 + 핵심 요구사항**으로 시작
- AI가 상세 스펙 초안을 작성하게 하고, 사람이 검토·수정
- 승인된 스펙을 `SPEC.md`로 저장 → 세션 간 지속적 참조점

**효과적인 시작 프롬프트:**
```
"You are an AI software engineer. Draft a detailed specification for [project X]
covering objectives, features, constraints, and a step-by-step plan."
```

**Plan Mode 활용 (Claude Code):**
- Shift+Tab → Plan Mode 진입 (읽기 전용, 코드 미작성)
- AI가 기존 코드베이스를 분석하고 스펙 초안 작성
- 아키텍처, 보안, 테스트 전략까지 검토
- **모호함이 없을 때까지 계획 정제 → 그 다음 실행**

### 원칙 2: 스펙을 구조화된 문서(PRD)로 작성

GitHub가 2,500개+ 에이전트 설정 파일을 분석한 결과, 효과적인 스펙이 다루는 **6가지 핵심 영역**:

| 영역 | 내용 |
|------|------|
| **Commands** | 실행 명령어를 먼저, 전체 명령어+플래그로 (`npm test`, `pytest -v`, `npm run build`) |
| **Testing** | 테스트 실행 방법, 프레임워크, 파일 위치, 커버리지 기대치 |
| **Project Structure** | 소스 코드/테스트/문서 위치를 명시 (`src/`, `tests/`, `docs/`) |
| **Code Style** | 3단락 설명보다 **실제 코드 스니펫 1개**가 효과적. 네이밍, 포맷 규칙 |
| **Git Workflow** | 브랜치 네이밍, 커밋 메시지 형식, PR 요구사항 |
| **Boundaries** | AI가 절대 건드리면 안 되는 것 — "Never commit secrets"가 가장 흔한 제약 |

**스펙 템플릿 예시:**
```markdown
# Project Spec: My team's tasks app

## Objective
- Build a web app for small teams to manage tasks...

## Tech Stack
- React 18+, TypeScript, Vite, Tailwind CSS
- Node.js/Express backend, PostgreSQL, Prisma ORM

## Commands
- Build: `npm run build` (컴파일 후 dist/ 출력)
- Test: `npm test` (Jest, 커밋 전 필수 통과)
- Lint: `npm run lint --fix` (ESLint 자동 수정)

## Project Structure
- `src/` – 애플리케이션 소스
- `tests/` – 유닛/통합 테스트
- `docs/` – 문서

## Boundaries
- ✅ Always: 커밋 전 테스트 실행, 네이밍 컨벤션 준수
- ⚠️ Ask first: DB 스키마 변경, 의존성 추가
- 🚫 Never: 시크릿 커밋, node_modules/ 수정, CI 설정 변경
```

**GitHub Spec Kit의 4단계 게이팅 워크플로우:**
1. **Specify**: 사용자 여정·경험·성공 기준 중심으로 스펙 작성
2. **Plan**: 기술 스택·아키텍처·제약사항 기반 상세 기술 계획 수립
3. **Tasks**: 스펙+계획을 독립적으로 구현·테스트 가능한 작은 단위로 분해
4. **Implement**: 에이전트가 태스크를 하나씩(또는 병렬로) 처리

### 원칙 3: 한 번에 하나의 태스크만

**"지시의 저주(Curse of Instructions)"**: 지시사항이 많아질수록 AI의 각 지시 준수율이 급격히 하락. GPT-4와 Claude조차도 동시에 많은 요구사항을 만족시키기 어렵다.

**해결 전략:**

- **스펙을 섹션별로 분리** — 백엔드/프론트엔드 스펙을 각각 분리, 필요한 부분만 제공
- **확장 목차(Extended TOC)** — 스펙을 섹션별 핵심 요약+참조 태그로 압축, 상세 내용은 필요시 로드
- **하위 에이전트(Subagents)** — 영역별로 전문 에이전트 분리 (DB 에이전트, API 에이전트, 프론트엔드 에이전트)
- **새 세션 시작** — 주요 기능 전환 시 컨텍스트 초기화, 핵심 전역 규칙만 다시 주입
- **코드 내 TODO 주석** — 미니 스펙 역할, 에이전트가 하나씩 채워나감

**단일 에이전트 vs 멀티 에이전트:**

| 항목 | 단일 에이전트 | 병렬/멀티 에이전트 |
|------|------------|----------------|
| 장점 | 간단한 설정, 낮은 오버헤드, 디버깅 용이 | 높은 처리량, 복잡한 의존성 처리, 도메인별 전문화 |
| 단점 | 큰 프로젝트에서 컨텍스트 과부하, 단일 장애점 | 조율 오버헤드, 공유 메모리 필요 |
| 적합 | 소~중형 프로젝트, 초기 프로토타입 | 대형 코드베이스, 코딩+테스트+리뷰 병렬 진행 |
| 팁 | 스펙 요약 활용, 태스크마다 컨텍스트 리프레시 | 초기엔 2-3개로 제한, MCP로 도구 공유 |

### 원칙 4: 품질 통제 가이드 + 전문 지식 주입

**3단계 경계 시스템 (2,500+ 분석에서 가장 효과적):**
- ✅ **Always do**: 묻지 않고 항상 실행 (테스트 실행, 네이밍 준수, 에러 로깅)
- ⚠️ **Ask first**: 인간 승인 필요 (DB 스키마 변경, 의존성 추가, CI/CD 설정 변경)
- 🚫 **Never do**: 절대 금지 (시크릿 커밋, node_modules 수정, 실패한 테스트 무단 삭제)

**자기 검증 패턴:**
```
(구현 후, 위의 요구사항 목록과 결과를 비교하고 모든 항목이 충족되었는지 확인하라.
충족되지 않은 항목이 있으면 목록으로 표시하라.)
```

**LLM-as-a-Judge:** 코드 스타일, 가독성, 아키텍처 패턴 준수 등 자동 테스트가 어려운 기준은 별도 에이전트로 리뷰:
```
"이 코드를 우리 스타일 가이드에 맞게 검토하고 위반 사항을 표시하라."
```

**도메인 지식 주입 예시:**
```
"라이브러리 X를 사용할 경우 버전 Y의 메모리 누수 이슈에 주의하라 (해결책: Z 적용)"
"모든 API 응답은 JSON. 에러 응답 예: {"error": "message"}"
```

### 원칙 5: 반복 루프 — 테스트 → 피드백 → 스펙 개선

**지속적 테스트:** 매 마일스톤마다 테스트 실행, 실패 결과를 다음 프롬프트에 피드백
```
에이전트 루프: 코드 작성 → 테스트 실행 → 실패 수정 → 반복
```

**스펙 자체도 반복:** 요구사항이 변경되거나 AI가 오해한 경우 스펙 즉시 업데이트 → 에이전트에게 변경사항 명시적 재동기화

**컨텍스트 관리 도구 활용:**
- RAG: 대형 스펙을 벡터 DB에 임베딩 → 현재 태스크에 관련된 섹션만 자동 로드
- MCP (Model Context Protocol): 현재 작업 기반으로 관련 컨텍스트 자동 주입

**병렬 에이전트 시 주의사항:**
- 동일 파일에 두 에이전트가 동시에 쓰지 않도록 태스크 독립성 보장
- 2-3개로 시작, 정신적으로 매우 피로한 작업

## 피해야 할 안티패턴

| 안티패턴 | 문제 |
|----------|------|
| **모호한 프롬프트** | "Build me something cool" → AI가 앵커 없음. "You are a test engineer who writes tests for React components, follows these examples, never modifies source code"처럼 구체적으로 |
| **요약 없는 과도한 컨텍스트** | 50페이지 문서를 통째로 넣어도 작동 안 함. RAG나 계층적 요약 사용 |
| **사람 검토 생략** | "내가 다른 사람에게 설명할 수 없는 코드는 커밋하지 않는다" (Willison 원칙) |
| **바이브 코딩 ≠ 프로덕션 엔지니어링** | 탐색용 프로토타입과 프로덕션 코드를 혼동하지 말 것 |
| **치명적 트리플 무시** | 속도(검토 전에 완성) + 비결정성(같은 입력, 다른 출력) + 비용(검증 생략) → 세 가지 모두 스펙과 리뷰 프로세스로 통제 |
| **6개 핵심 영역 누락** | Commands, Testing, Structure, Style, Git, Boundaries — 하나라도 빠지면 에이전트가 허둥댐 |

## 요약 체크리스트

- [ ] 고수준 목표로 시작하고 AI가 상세 스펙 초안 작성하게 하기
- [ ] SPEC.md를 버전 관리에 포함, 세션 간 지속적 참조점으로 활용
- [ ] 6개 핵심 영역 (Commands, Testing, Structure, Style, Git, Boundaries) 모두 포함
- [ ] 3단계 경계 시스템 (✅ Always / ⚠️ Ask first / 🚫 Never) 명시
- [ ] 한 번에 하나의 태스크만 — 큰 스펙은 분리하여 관련 섹션만 주입
- [ ] 각 태스크 후 자동 테스트 + 스펙 준수 여부 확인
- [ ] 스펙을 살아있는 문서로 유지 — 결정사항 즉시 반영
