## 1. CLAUDE.md — 에이전트의 헌법

### 기본 원칙
- **팀 전체가 공유하는 단일 CLAUDE.md** — git에 체크인, 팀이 주 단위로 여러 번 기여 *(#25570)*
- **Claude가 잘못된 행동을 할 때마다 추가** — 같은 실수 방지, 누적 학습 *(#25570)*
- **코드 리뷰 시 @.claude 태그로 PR의 일부로 CLAUDE.md 업데이트** *(#25570)*
- **AGENTS.md와 동기화** — 다른 AI IDE와 호환성 유지 *(#24099)*

### 작성 원칙 (안티패턴 포함) *(#24099)*

| 원칙 | 설명 |
|------|------|
| **가드레일로 시작** | Claude가 잘못하는 부분 기반으로 소규모 문서화 시작. 매뉴얼 아님 |
| **@-파일 문서화 금지** | 다른 문서를 @-언급하면 매 실행마다 전체 파일이 컨텍스트 윈도우에 로드 |
| **"절대 안 됨"만 말하지 말 것** | 항상 대안 제시. 부정형만 쓰면 에이전트가 막힘 |
| **간결하게 유지** | 도구를 간결하게 설명 못하면 CLAUDE.md 준비 미완료. 짧게 유지하는 것이 코드베이스 단순화를 위한 강제 함수 |
| **"절대 --foo-bar 플래그 사용 금지" 회피** | 에이전트가 해당 플래그가 필요하다 생각할 때 막힘 |

### 권장 구조 예시 *(#24099)*
```markdown
# Monorepo

## Python
- Always ...
- Test with: pytest -v
... (80% 사용 사례에 집중한 10개 불릿)

## <내부 도구명>
- Always ...
- Never X, prefer Y
- 복잡한 사용법이나 FooBarError 발생 시 path/to/_docs.md 참조
```

- 엔터프라이즈 모노레포: 13KB 규모 엄격 관리 (25KB까지 확장 가능)
- 엔지니어 30% 이상이 사용하는 도구만 문서화
- 각 내부 도구 문서화에 최대 토큰 수 할당 ("광고 공간" 판매 방식)

### Git Worktrees로 병렬 브랜치 작업 *(#26526)*

## 5. 컨텍스트 관리

### 3가지 주요 워크플로우 *(#24099)*

| 방법 | 언제 사용 | 방법 |
|------|-----------|------|
| `/compact` 회피 | — | 자동 압축은 불투명, 오류 발생, 최적화 부족 → 최대한 사용 지양 |
| `/clear` + `/catchup` | 단순 재시작 | `/clear`로 상태 초기화 → 커스텀 `/catchup`으로 git 브랜치 변경 파일 읽기 |
| "Document & Clear" | 대규모 작업 | Claude가 계획·진행상황을 .md에 덤프 → `/clear` → 새 세션에서 .md 읽고 계속 |

### 컨텍스트 팁
- **`/context`** 명령으로 200k 토큰 윈도우 사용 현황 확인 *(#24099)*
- 모노레포 새 세션 기본 ~20k 토큰(10%) 소비, 나머지 180k는 작업용
- **선제적 컨텍스트 압축** — 윈도우가 차기 전에 미리 압축 *(#26526)*
- **`#`으로 메모리 빠르게 추가** — 알아서 가장 적절한 메모리 파일에 넣음 *(#22034)*

> **"이 피드백 루프가 있으면 최종 결과물 품질이 2~3배 향상"** *(#25570)*

### 검증 방식
- **Claude Chrome 확장 프로그램** — 브라우저를 열고 UI를 직접 테스트, 코드 작동 + UX 좋을 때까지 반복 *(#25570)*
- **bash 명령 실행** — 단순하지만 강력
- **테스트 스위트 실행** — 자동화된 검증
- **브라우저/폰 시뮬레이터에서 앱 테스트**

### Hooks로 검증 강제 *(#24099)*

| Hook 유형 | 전략 |
|-----------|------|
| **커밋 단계 차단 Hook (Block-at-Submit)** | PreToolUse Hook으로 `git commit` 래핑 → `/tmp/agent-pre-commit-pass` 파일 확인 → 없으면 커밋 차단 → "테스트-수정" 루프 강제 |
| **힌트 Hook** | 비차단, 에이전트가 차선책 수행 시 피드백 제공 |
| **쓰기 단계 차단 회피** | 계획 중간에 에이전트 차단 시 혼란 유발 → 커밋 단계에서 최종 확인이 훨씬 효과적 |

### PostToolUse 훅으로 코드 포매팅 *(#25570)*

## 8. 서브에이전트 & Task()

### 이론 vs 실전 문제 *(#24099)*

**이론상 장점**: 컨텍스트 절약
- N개 작업 = 메인 윈도우에서 (X+Y+Z)×N 토큰
- Subagent: (X+Y)×N을 특화 에이전트에 위임, 최종 Z 토큰만 반환

**실전 문제점**:
1. **컨텍스트 게이트키핑** — PythonTests Subagent 생성 시 메인 에이전트에서 테스트 컨텍스트 숨김 → 전체적 추론 불가능
2. **인간 워크플로우 강제** — Claude를 경직된 인간 정의 워크플로우로 강제

### 권장: `Task()` 빌트인 기능 *(#24099)*
- CLAUDE.md에 모든 핵심 컨텍스트 배치
- 메인 에이전트가 **범용 에이전트 복사본**에 작업 위임 시점과 방법을 스스로 결정
- 컨텍스트 절약 이점은 유지하면서 단점 제거
- **"Master-Clone" 아키텍처** — Lead-Specialist 모델보다 강력히 선호

### Boris Cherny의 서브에이전트 *(#25570)*
- **code-simplifier** — Claude 작업 완료 후 코드 단순화
- **verify-app** — Claude Code 엔드투엔드 테스트를 위한 상세 지침

## 10. MCP (Model Context Protocol)

### Skills 등장 후 MCP의 새로운 역할 *(#24099)*

| 기존 (잘못된 방식) | 새로운 역할 |
|-------------------|-------------|
| REST API 미러링하는 수십 개 도구 (read_thing_a, update_thing_c...) | 몇 가지 강력한 고수준 도구를 제공하는 단순·안전한 게이트웨이 |
| 에이전트를 위한 현실 추상화 | 인증, 네트워킹, 보안 경계 관리 후 방해하지 않기 |

### 권장 MCP 패턴 *(#24099)*
```
download_raw_data(filters…)
take_sensitive_gated_action(args…)
execute_code_in_environment_with_state(code…)
```

- 현재 사용 중인 유일한 MCP: **Playwright** (복잡하고 상태 저장 환경이라 타당)
- 모든 무상태 도구 (Jira, AWS, GitHub)는 단순 CLI로 마이그레이션

### Boris Cherny의 MCP 도구 통합 *(#25570)*
- Slack 검색 및 게시 (MCP 서버)
- BigQuery 쿼리 실행 (bq CLI)로 분석 질문 답변
- Sentry에서 에러 로그 가져오기
- `.mcp.json`에 체크인, 팀과 공유

## 11. Git & GitHub 통합

### GitHub Action (GHA) *(#24099)*
- 단순 개념(GHA에서 Claude Code 실행)이지만 커스터마이즈 가능성이 강력함의 원천
- 전체 컨테이너와 환경 제어 → 데이터 접근성 향상
- Cursor 백그라운드 에이전트보다 훨씬 강력한 샌드박싱·감사 제어
- Hook, MCP 같은 모든 고급 기능 지원

**활용 사례:**
- Slack, Jira, CloudWatch 알림에서 PR 트리거 → 완전히 테스트된 PR 반환
- GHA 로그 = 전체 에이전트 로그 → 공통 실수·bash 오류·정렬되지 않은 관행 발견
- 플라이휠: 버그 → CLAUDE.md/CLI 개선 → 더 나은 에이전트
```bash
$ query-claude-gha-logs --since 5d | claude -p "see what the other claudes were getting stuck on and fix it, then put up a PR"
```

### GitHub 통합 *(#22034)*
- **`/install-github-app`** — PR 리뷰 기능 (로직 에러, 보안 이슈도 잘 찾음)
- 기본 리뷰 프롬프트는 별로 → `claude-code-review.yml`에서 수정 필요

## 14. Claude Code SDK *(#24099)*

- 대화형 CLI일 뿐 아니라 **완전히 새로운 에이전트 구축용 SDK**
- LangChain/CrewAI보다 기본 에이전트 프레임워크로 우선 고려

**3가지 주요 사용 방식:**
1. **대규모 병렬 스크립팅** — 대규모 리팩토링·마이그레이션에 병렬 `claude -p` 호출
2. **내부 채팅 도구 구축** — 비기술 사용자를 위해 복잡한 프로세스를 채팅으로 래핑
3. **신속한 에이전트 프로토타이핑** — 커스텀 CLI나 MCP 아이디어를 빠르게 검증

## 15. settings.json 핵심 설정 *(#24099)*

| 설정 | 용도 |
|------|------|
| `HTTPS_PROXY/HTTP_PROXY` | 디버깅용 — 원시 트래픽 검사로 Claude가 전송하는 정확한 프롬프트 확인 |
| `MCP_TOOL_TIMEOUT/BASH_MAX_TIMEOUT_MS` | 값 증가 — 기본 타임아웃이 종종 너무 보수적 |
| `ANTHROPIC_API_KEY` | 엔터프라이즈 API 키 사용 → "좌석당" 라이선스에서 "사용량 기반" 가격으로 전환 |

## 16. UI & 단축키 팁 *(#22034, #26526)*

| 팁 | 내용 |
|----|------|
| **줄바꿈** | `Shift+Enter` 안 됨 → `/terminal-setup` 하면 해결 (또는 `Ctrl+J`) |
| **파일 드래그** | Shift 누른 채로 드래그 해야 참조 (그냥 드래그하면 파일 열림) |
| **중지** | `Ctrl+C` 아니라 `Esc` |
| **클립보드 이미지** | Mac에서도 `CMD+V` 아닌 `CTRL+V` |
| **이전 세션** | `↑` 화살표 → 이전 챗. `Esc` 두 번 → 모든 메시지 보기 |
| **Vim 모드** | `/vim-mode`로 프롬프트 Vim 편집 *(#26526)* |

## 17. Builder.io 특수 케이스 *(#22034)*

- **18,000줄짜리 단일 React 컴포넌트** — 제대로 다룬 AI가 없었는데 Claude Code는 해냄

- 큐 시스템: 작업 중 빼먹은 것을 하나씩 추가해두면 알아서 처리, 피드백 필요하면 Claude가 되물음
